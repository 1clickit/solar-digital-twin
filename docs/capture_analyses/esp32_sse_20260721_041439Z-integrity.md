# ESP32 Endurance Capture Integrity Review

## Result

Capture `esp32_sse_20260721_041439Z` is accepted as **PASS WITH ONE
QUALIFICATION**. Its immutable raw, retained, and manifest evidence is
internally consistent and suitable for later forensic analysis. This review
did not assess the semantic or physical plausibility of telemetry values.

The single qualification is operational: systemd ignored `RuntimeMaxSec`
because transient unit `esp32-forensic-overnight-20260721.service` used
`Type=oneshot`. Automatic completion therefore depended on the collector's own
finite `--duration`. That duration completed normally and does not invalidate
or weaken the captured telemetry contents. A future endurance launch requires
a separate review if an independent host-enforced runtime ceiling is desired.

## Purpose and scope

This was a finite post-hardening endurance test of the ESP32 forensic probe and
installed `solardt` collection path across nighttime and sunrise. It was not a
test of the synthetic SolarAssistant adapter. Integrity review was bounded,
read-only, and did not modify evidence, runtime, services, permissions, the
device, networking, or repository source.

Persistent ESP32 operation remains unauthorized. This accepted finite capture
does not authorize a timer, enablement, persistent service operation, another
endurance run, firmware or device change, or network action.

ESP32 timer remained. The ordinary static ESP32 unit remained inactive/dead
with zero restarts and no persistent activation. The compact journal reported
approximately 1 minute 55.887 seconds CPU, 18.1 MB peak memory, and zero swap.

Systemd also reported that `RuntimeMaxSec` had no effect with `Type=oneshot`.
The collector's finite application duration still stopped normally at the
planned endpoint; the missing independent host cutoff is the review's sole
qualification.
